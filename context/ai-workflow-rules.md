# Development Workflow

## Approach

Build this project incrementally using a spec-driven workflow. Context files define what to build, how to build it, and what the current state of progress is. Always implement against these specs — do not infer or invent behavior from scratch.

The project is a full-stack employee management system for a single company. The MVP should stay simple, but every implementation step must preserve the architecture foundations needed for future growth.

Primary context files:

- `project-overview.md` — product scope, users, features, and success criteria.
- `architecture-context.md` — stack, boundaries, storage, auth, deployment, performance, and invariants.
- `ui-context.md` — theme tokens, visual references, layout patterns, and styling rules.
- `code-standards.md` — implementation rules and coding conventions.
- `features/current-feature.md` — active feature scope, checklist, open questions, and notes.
- `features/feature-history.md` — concise history of completed features.

## Feature Branch Workflow

- Before starting a new feature, create a new feature branch from `main`; if the user explicitly says not to create a branch, skip this workflow and follow the user's instructions.
- After agreeing on the feature scope with the user and before implementation, fill the placeholders in `features/current-feature.md`, then ask whether to start implementation.
- After implementation is complete and the user asks to commit and merge into `main`, reset `features/current-feature.md` back to placeholders and add a concise entry to `features/feature-history.md`.

## Scoping Rules

- Work on one feature unit or subsystem at a time.
- Prefer small, verifiable increments over large speculative changes.
- Do not combine unrelated system boundaries in a single implementation step.
- Keep MVP behavior simple unless a future-proof foundation is cheap and clearly useful.
- Security-sensitive behavior must be implemented on the backend, not only hidden in the frontend.
- Performance must be considered before implementation, especially for list pages, database queries, file handling, and authenticated requests.

## When To Split Work

Split an implementation step if it combines:

- Frontend UI changes and backend API changes that cannot be verified independently.
- Auth/session logic and unrelated feature business logic.
- RBAC/policy changes and broad data model changes.
- Multiple unrelated API route groups.
- Database migrations and complex UI behavior.
- File upload/storage work and unrelated employee/attendance/leave logic.
- Behavior that is not clearly defined in the context files.

If a change cannot be verified end to end quickly, the scope is too broad — split it.

Examples of good feature units:

- Project setup and tooling.
- Auth database models and session middleware.
- Admin employee list API with pagination.
- Employee list page wired to TanStack Query.
- Payslip upload to Cloudflare R2.
- Leave request approval flow.

Examples of overly broad feature units:

- Complete auth, employees, attendance, leave, and payslips in one step.
- Full admin dashboard with all APIs and UI pages.
- Payroll automation plus payslip storage plus attendance calculations.

## Handling Missing Requirements

- Do not invent product behavior that is not defined in the context files.
- If a requirement is ambiguous, resolve it in the relevant context file before implementing.
- If a requirement is missing, add it as an open question in `features/current-feature.md` before continuing.
- If implementation must proceed before the question is answered, choose the simplest reversible option and record the assumption in `features/current-feature.md`.
- Do not add third-party services unless the architecture context explicitly allows them.
- Do not add public registration, public company onboarding, or unrestricted public-demo uploads unless explicitly requested.

Common requirements that must be clarified before implementation if they appear:

- Working hours, lateness rules, and overtime rules.
- Whether employees may edit profile fields themselves.
- Whether payslips are uploaded manually or generated by the system.
- Whether leave balances are manually managed or automatically accrued.
- Whether admin-created employee accounts use invitation emails.
- Whether a public demo deployment should disable uploads, quota-limit uploads, or clean up uploaded demo files.
- Whether attendance clock-in/out needs geolocation, device checks, or IP restrictions.

## Protected Foundation Components

Do not modify generated third-party foundation components unless explicitly instructed.

This includes:

- shadcn/ui components under `client/src/shared/components/ui/*` or equivalent generated UI directory.
- Third-party library internals.
- Generated Prisma client output.
- Build tool generated files.

Project-specific styling, layout changes, and feature logic must be implemented in app-level or feature-level components instead of modifying foundation components.

Only modify protected files when a task explicitly requires it.

## Keeping Docs In Sync

Update the relevant context file whenever implementation changes:

- System architecture or boundaries.
- Storage model decisions.
- Auth/session/RBAC behavior.
- Code conventions or folder structure.
- UI theme tokens, visual references, or global styling conventions.
- Feature scope.
- Deployment assumptions.
- Performance constraints.
- Known risks or failure modes.

Progress state must reflect the actual state of the implementation, not the intended state.

Documentation update rules:

- Update `features/current-feature.md` while a branched feature is being planned or implemented.
- Move a concise summary into `features/feature-history.md` only after the feature works end to end and the user asks to commit and merge.
- Update `architecture-context.md` when a decision affects multiple features or system boundaries.
- Update `code-standards.md` when a coding convention becomes project-wide.
- Update `ui-context.md` when theme tokens, layout rules, or visual references change.
- Do not store long implementation logs in the context files; keep entries useful and concise.

## Before Moving To The Next Unit

1. The current unit works end to end within its defined scope.
2. No invariant defined in `architecture-context.md` was violated.
3. Authentication, authorization, and ownership checks were verified where relevant.
4. List endpoints use pagination or have an explicit reason not to.
5. Database queries avoid obvious overfetching and N+1 patterns.
6. Sensitive data is not logged or exposed in API responses.
7. `features/current-feature.md` reflects completed work until the user asks to commit and merge.
8. On user request to commit and merge, `features/current-feature.md` is reset to placeholders and `features/feature-history.md` receives a concise entry.
9. Public registration remains absent, and demo-mode storage guardrails remain intact when relevant.
